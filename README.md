## 1. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

Разработанная система представляет собой дискретно-событийную имитационную модель для анализа работы трамвайной сети. Модель построена на базе библиотеки SimPy и поддерживает одновременную симуляцию нескольких маршрутов в прямом и обратном направлениях.

### Основные задачи модели:

- Обработка реальных данных датчиков пассажиропотока (очистка, интерполяция, восстановление геометрии маршрутов)
- Назначение глобально уникальных идентификаторов остановок с сохранением читаемых локальных номеров
- Моделирование движения трамваев по заданным маршрутам с учётом графика движения
- Симуляция пассажиропотока с учётом времени суток и популярности остановок
- Расчёт ключевых показателей эффективности (загруженность, время ожидания, пропускная способность)
- Подсчёт транспортных показателей: трамвай-километры и пассажиро-километры
- Визуализация результатов для анализа узких мест системы

Модель позволяет тестировать различные конфигурации расписания и парка транспортных средств без проведения дорогостоящих натурных экспериментов.

## 3. ЛОГИКА РАБОТЫ МОДЕЛИ

### 3.1 Инициализация системы

При запуске мультимаршрутной симуляции происходит:

1. Загрузка всех конфигурационных JSON-файлов указанных маршрутов
2. Сборка глобального реестра остановок по уникальным `global_stop_id`
3. Создание отдельного экземпляра `RouteSimulation` для каждого маршрута/направления
4. Формирование парка трамваев (`tram_count`) для каждого маршрута
5. Параллельный запуск всех маршрутов в единой среде SimPy

Каждый маршрут полностью изолирован по ресурсам, но разделяет общие часы симуляции.

### 3.2 Генерация пассажиров

Количество прибывающих пассажиров рассчитывается на основе:

- **Базовой интенсивности остановки** — задаётся для каждого часа отдельно, извлекается из реальных данных датчиков
- **Времени с последнего трамвая** — чем дольше не было трамвая, тем больше накопилось пассажиров

Для имитации случайности применяется нормальное распределение как аппроксимация распределения Пуассона для больших интенсивностей.

**Формула:**

```

новых_пассажиров = N(μ = интенсивность × Δt, σ = √μ)
где N — нормальное распределение

```

### 3.3 Жизненный цикл трамвая

Каждый трамвай проходит следующий цикл:

1. **Ожидание** — в пуле до момента выхода. Трамваи выходят только в рабочие часы (`operation_start_hour` – `operation_end_hour`)
2. **Выход на маршрут** — согласно расписанию (`bus_interval`)
3. **Движение по маршруту** — последовательный проезд всех остановок в порядке `stop_ids`
4. **Разворот** — пауза на конечной остановке (`turnaround_time`)
5. **Возврат на базу** — трамвай освобождается и возвращается в пул
6. **Повторный выпуск** — при необходимости в рабочие часы

Прямое и обратное направления — это два независимых маршрута с разными конфигами (`_fwd` и `_bwd`).

### 3.4 Логика остановки

При прибытии трамвая на остановку:

1. **Высадка пассажиров:**
   - Базовый процент высадки (20%)
   - Удвоение на популярной остановке (`peak_stop`)
   - Увеличение по мере приближения к концу маршрута
   - Полная высадка на конечной остановке
2. **Генерация новых пассажиров** по интенсивности и времени ожидания
3. **Посадка:** `min(ожидающих, свободных_мест)`
4. **Учёт времени:** фиксация ожидания, обновление времени обслуживания, логирование загруженности

### 3.5 Расчёт времени движения

Время проезда между остановками зависит от:

- Расстояния (вычисляется из медианного времени хода по реальным логам)
- Базовой скорости потока (`flow_speed`)
- Загруженности дорог в текущий час (`road_loads`)
- Случайного отклонения ±5% (имитация светофоров, маневрирования)

**Формула:**

```

скорость = базовая_скорость × (1 − загруженность) × случайность
время = расстояние / скорость + время_разгона

```

В часы пик (загруженность 0.9) скорость падает до ~10% от номинальной.

## 4. МОДУЛИ И ИХ ФУНКЦИИ

### 4.1 models/route.py

Ядро симуляции одного маршрута одного направления.

**Основные классы:**

- `Stop` — модель остановки с очередью пассажиров
  - `get_new_passengers()` — генерация прибывших пассажиров
  - `record_waiting()` — запись состояния для построения графиков
- `Tram` — модель трамвая
  - `board_passengers()` — посадка с учётом вместимости
  - `alight_passengers()` — высадка по логике маршрута
  - `log_stop_event()` — детальное логирование для CSV
- `RouteSimulation` — управляющий класс маршрута
  - `tram_generator()` — выпуск трамваев по расписанию с учётом рабочих часов
  - `tram_process()` — жизненный цикл трамвая
  - `arrive_at_stop()` — обработка прибытия, расчёт загруженности
  - `calculate_travel_time()` — расчёт времени в пути

### 4.2 simulation/multi_route.py

Оркестратор мультимаршрутной симуляции.

- `MultiRouteSimulation` — запускает все маршруты в единой среде SimPy
- Ведёт глобальный реестр остановок (`global_stop_id → Stop`)
- Агрегирует результаты по всем маршрутам после завершения симуляции
- Передаёт управление `logger.py` и `visualization.py` для каждого маршрута отдельно

### 4.3 simulation/runner.py

CLI-точка входа.

- Принимает номера маршрутов через `--routes`
- Автоматически находит `fwd`/`bwd` конфиги по паттерну `route_{id}_{dir}_config.json`
- При отсутствии любого конфига — выводит список отсутствующих файлов и прерывает запуск
- Поддерживает флаги `--no-plots`, `--no-logs`, `--configs-dir`

### 4.4 visualization.py

Модуль построения аналитических графиков. Автоматически определяет реальные ключи остановок (`global_stop_id`) и отображает их как локальные номера 1..N для читаемости.

- `plot_waiting_passengers()` — динамика очередей на каждой остановке
- `plot_waiting_by_hour()` — среднее количество ожидающих по часам
- `plot_heatmap()` — тепловая карта загруженности (остановки × часы)
- `plot_utilization()` — загруженность трамваев и количество рейсов
- `plot_tram_utilization_by_hour()` — почасовая динамика загруженности парка с отрисовкой часов пик

Графики сохраняются в PNG с разрешением 150 DPI. Папка с графиками создаётся для каждого маршрута/направления отдельно.

### 4.5 logger.py

Система детального логирования:

- Создаёт отдельный CSV для каждого трамвая с полной историей остановок
- Формирует сводную таблицу `trams_summary.csv`
- Записывает: время, остановку (локальный номер), направление, ожидавших, высадку, посадку, пассажиров в салоне, загруженность (%)
- Поддерживает поле `route_id` для идентификации маршрута в мультимаршрутном запуске

### 4.6 clear_script.ipynb

Ноутбук для конвертации реальных Excel-выгрузок датчиков пассажиропотока в конфиги модели. Покрывает маршруты 20, 48, 55 в обоих направлениях.

- Фильтрация «сломанных» датчиков (отсечение выбросов >150 пасс., удаление нулевых рейсов)
- Восстановление геометрии маршрута: построение эталонной последовательности остановок и расчёт расстояний на основе медианного времени хода
- Назначение `global_stop_id` с префиксами по маршруту (20→20001..., 48→48001..., 55→55001...)
- Поддержка `STOP_ID_OVERRIDES` для физически совпадающих остановок разных маршрутов
- Расчёт интенсивности `intensity` с усреднением по числу валидных дней (мин. `MIN_DAYS = 3`)
- Интерполяция пропущенных часов (линейная внутри дня, затухание на краях)
- Генерация готовых `.json` файлов в папку `configs/`

### 4.7 generate_config.py

Генератор полностью синтетических конфигураций:

- Быстрое создание тестовой среды без реальных данных
- Экспортирует JSON в компактном читаемом формате
- Поддерживает `route_id`, `stop_ids`, `operation_start_hour = 5`

```json
{

  "stop_number": 34,
  "distance": [, , ],

  "road_loads": [[0, 0.1], [5, 0.2], [6, 0.4], [7, 0.7], [8, 0.9],
                 [10, 0.6], [16, 0.7], [17, 0.9], [20, 0.4], [23, 0.1]],
  "flow_speed": 15,
  "peak_stop": 20017,
  "tram_capacity": 150,

  "operation_end_hour": 24,

  "stop_time": 1.0,
```

### Описание полей:

| Параметр | Тип | Описание |
| :-- | :-- | :-- |
| `route_id` | String | Идентификатор маршрута/направления, например `"20_fwd"` |
| `stop_ids` | Array | Упорядоченный список глобальных ID остановок маршрута |
| `stop_number` | Integer | Количество остановок (должно совпадать с длиной `stop_ids`) |
| `distance` | Array | `[[global_stop_id, метров_от_предыдущей], ...]`, первая остановка = 0 |
| `intensity` | Array | `[[global_stop_id, час, пассажиров_в_час], ...]` — покрывает все 24 часа |
| `bus_interval` | Array | `[[час_начала, минут_между_трамваями], ...]` — с 5:00 до конца работы |
| `road_loads` | Array | `[[час, коэффициент_0–1], ...]` — влияние загруженности дорог на скорость |
| `flow_speed` | Integer | Базовая скорость (км/ч) при нулевой загруженности дорог |
| `peak_stop` | Integer | `global_stop_id` самой загруженной остановки |
| `tram_capacity` | Integer | Вместимость трамвая (пасс.) |
| `tram_count` | Integer | Физическое количество трамваев в парке |
| `operation_start_hour` | Integer | Час начала выпуска трамваев (обычно `5`) |
| `operation_end_hour` | Integer | Час окончания работы (обычно `24`) |
| `simulation_hours` | Integer | Длительность моделирования в часах |
| `acceleration_time` | Float | Время разгона/торможения на остановке (мин) |
| `stop_time` | Float | Базовое время стоянки (мин) |
| `turnaround_time` | Float | Время разворота на конечной (мин) |
| `target_utilization` | Float | Целевая загруженность вагона (доля, обычно 0.75) |
| `random_seed` | Integer | Зерно генератора случайных чисел для воспроизводимости |

### Важные замечания:

**1. Глобальные ID остановок:**

- Каждая остановка имеет уникальный `global_stop_id` в формате `{маршрут}{локальный_номер}`: маршрут 20 → 20001..20999, маршрут 48 → 48001..., маршрут 55 → 55001...
- В графиках и логах отображаются **локальные номера 1..N** для читаемости
- Если две остановки физически совпадают между маршрутами, их `global_stop_id` задаётся вручную через `STOP_ID_OVERRIDES` в `clear_script.ipynb`

**2. Рабочие часы:**

- `operation_start_hour = 5` — первые трамваи выходят в 5:00
- Вне рабочих часов трамваи **не выходят**, но пассажиры **накапливаются** на остановках

**3. Интенсивность пассажиров:**

- Задаётся для **всех 24 часов** (включая ночное время — нулевые значения)
- `clear_script.ipynb` автоматически интерполирует пропуски и сглаживает краевые часы

- Список найденных конфигов и подтверждение запуска

## 7. ЗАПУСК И ИСПОЛЬЗОВАНИЕ

### 7.1 Установка зависимостей

```bash
pip install -r requirements.txt
```


### 7.2 Генерация конфигураций из реальных данных

Поместите Excel-файлы с датчиков в папку `data_sensors/` и запустите все ячейки `clear_script.ipynb` через Jupyter Notebook/Lab или VS Code. Готовые конфиги для всех маршрутов и направлений появятся в `configs/`.

### 7.3 Генерация синтетических конфигураций

```bash
python generate_config.py
```


### 7.4 Запуск симуляции

**Все три маршрута (fwd + bwd каждого — итого 6 конфигов):**

```bash
python -m simulation.runner --routes 20 48 55
```

**Один маршрут:**

```bash
python -m simulation.runner --routes 20
```

**Быстрый режим без графиков:**

```bash
python -m simulation.runner --routes 20 48 55 --no-plots
```

**Без детальных CSV-логов:**

```bash
python -m simulation.runner --routes 20 48 55 --no-logs
```

**Нестандартная папка с конфигами:**

```bash
python -m simulation.runner --routes 20 48 --configs-dir data/configs
```

Runner автоматически ищет файлы по паттерну `route_{id}_fwd_config.json` / `route_{id}_bwd_config.json`. Если хотя бы один файл отсутствует — выводится список недостающих и симуляция не запускается.

### 7.5 Анализ результатов

Результаты сохраняются в `outputs/run_ГГГГ-ММ-ДД_ЧЧ-ММ-СС/`. Каждый запуск создаёт отдельную папку с timestamp — это позволяет сравнивать разные конфигурации между собой.
