## 1. ОБЩЕЕ ОПИСАНИЕ СИСТЕМЫ

Разработанная система представляет собой дискретно-событийную имитационную модель для анализа работы трамвайной сети. Модель построена на базе библиотеки SimPy и поддерживает одновременную симуляцию нескольких маршрутов в прямом и обратном направлениях в рамках **единого общего парка трамваев**.

### Основные задачи модели:

- Обработка реальных данных датчиков пассажиропотока (очистка, интерполяция, восстановление геометрии маршрутов)
- Назначение глобально уникальных идентификаторов остановок с сохранением читаемых локальных номеров
- Моделирование движения трамваев по реальному расписанию с отслеживанием отклонений
- Симуляция пассажиропотока с учётом времени суток и популярности остановок
- Расчёт ключевых показателей эффективности (загруженность, время ожидания, пропускная способность)
- Подсчёт транспортных показателей: трамвай-километры и пассажиро-километры
- Визуализация результатов для анализа узких мест системы

Модель позволяет тестировать различные конфигурации расписания и размера парка без проведения дорогостоящих натурных экспериментов. Является основой для оптимизации параметров методом NSGA-II.

## 3. ЛОГИКА РАБОТЫ МОДЕЛИ

### 3.1 Инициализация системы

При запуске мультимаршрутной симуляции происходит:

1. Загрузка конфигурационных JSON-файлов для каждого маршрута (fwd + bwd)
2. Сборка глобального реестра остановок по уникальным `global_stop_id`
3. Создание `TramPair` для каждого маршрута — пары fwd/bwd, связанных общим пулом трамваев
4. Формирование парка: `tram_counts[i]` трамваев на пару, глобально уникальные ID
5. Параллельный запуск всех маршрутов в единой среде SimPy

### 3.2 Архитектура TramPair

Каждая пара fwd/bwd работает через три SimPy Store:

```

pool (депо) → [fwd прогон] → fwd_done → [разворот] → [bwd прогон] → bwd_done → [отдых] → pool

```

- `pool` — свободные трамваи, ожидающие выезда
- `fwd_done` — трамваи завершившие fwd, ожидающие разворота (`turnaround_time`)
- `bwd_done` — трамваи завершившие bwd, ожидающие отдыха водителя (`min_rest_time`)

Разворот и отдых — отдельные SimPy-процессы внутри `TramPair`.

### 3.3 Диспетчеризация по расписанию

Трамваи выпускаются строго по расписанию из JSON-конфига (`schedule`). Каждый рейс (`TripSchedule`) содержит плановые времена прибытия на каждую остановку.

**Ключевое правило:** если в момент планового отправления в пуле нет свободного трамвая — рейс **пропускается** с записью в лог. Это исключает аномальные задержки 400+ минут, которые возникают при ожидании освободившегося трамвая.

### 3.4 Генерация пассажиров

Количество прибывающих пассажиров рассчитывается на основе:

- **Базовой интенсивности остановки** — задаётся для каждого часа отдельно, извлекается из реальных данных датчиков
- **Времени с последнего трамвая** — чем дольше не было трамвая, тем больше накопилось пассажиров

**Формула:**

```

новых_пассажиров = N(μ = интенсивность × Δt, σ = √μ)
где N — нормальное распределение

```

### 3.5 Логика остановки

При прибытии трамвая на остановку:

1. **Высадка пассажиров:**
   - Базовый процент высадки (20%)
   - Удвоение на популярной остановке (`peak_stop`)
   - Увеличение по мере приближения к концу маршрута
   - Полная высадка на конечной остановке
2. **Генерация новых пассажиров** по интенсивности и времени ожидания
3. **Посадка:** `min(ожидающих, свободных_мест)`
4. **Логирование:** фиксируются `trip_id`, плановое время (`planned_time`), фактическое время и отклонение (`delay_min`)

## 3.6 Расчёт времени движения

Время проезда между остановками зависит от:

- Расстояния (вычисляется из медианного времени хода по реальным логам датчиков)
- Базовой скорости потока (`flow_speed`)
- Загруженности дорог в текущий час (`road_loads`)
- Случайного отклонения ±5% (имитация светофоров, маневрирования)

**Формула:**

```
скорость = базовая_скорость × (1 − загруженность) × случайность
время = расстояние / скорость + время_разгона
```

В часы пик (загруженность 0.9) скорость падает до ~10% от номинальной.

### Калибровка расписания

Плановые времена прибытия в расписании генерируются с той же физической моделью, что и симуляция, чтобы исключить систематическое смещение `delay_min`. Расписание учитывает:

- **Эффективную скорость** — `flow_speed × (1 − avg_load)`, где `avg_load` — средняя дневная загруженность из `road_loads` (~0.53, итоговая скорость ~7 км/ч)
- **Время стоянки** — фиксированное `stop_time = 1.0` мин
- **Среднее время посадки/высадки** — `avg_boarding_time = 0.9` мин (18.4 пасс. × 0.05 мин/пасс., из данных симуляции)
- **Буфер неравномерности** — `schedule_buffer = 1.5` мин на перегон (коэффициент неравномерности движения: светофоры, приоритет других ТС)

При адекватном размере парка (без массовых пропусков рейсов) среднее отклонение `delay_min` находится в диапазоне ±5 мин.

***

## 3.7 Отслеживание отклонений от расписания

На каждой остановке вычисляется и записывается `delay_min = actual_time − planned_time`. Знак определяет характер отклонения:

- `+` опоздание (трамвай задержался)
- `−` опережение (трамвай приехал раньше графика)

Отклонения накапливаются в `tram.stats.schedule_deviations` и экспортируются в отдельный `schedule_deviations.csv`.

### Интерпретация при малом парке

При недостаточном количестве трамваев часть рейсов **пропускается** (нет свободного вагона в момент планового отправления). Пропущенные рейсы не попадают в `schedule_deviations.csv`. Это создаёт эффект смещения выборки: в статистику входят преимущественно ранние рейсы, когда пул полон и трамваи едут без задержек — среднее `delay_min` уходит в минус. Такое поведение является **корректным** и отражает реальную деградацию системы при нехватке подвижного состава.

## 4. МОДУЛИ И ИХ ФУНКЦИИ

### 4.1 models/tram.py

Модель трамвая и его статистики. Не зависит от SimPy — чистые данные и логика.

- `TramStats` — датакласс: `passengers_served`, `total_trips`, `utilization_history`, `stop_log`, `schedule_deviations`
- `Tram.board_passengers()` — посадка с учётом вместимости
- `Tram.alight_passengers()` — высадка по логике маршрута (процент + конечная)
- `Tram.log_stop_event()` — добавляет запись в `stop_log` с `trip_id`, `planned_time`, `delay_min`
- `Tram.log_schedule_deviation()` — добавляет запись в `schedule_deviations`

### 4.2 models/stop.py

Модель остановки с очередью пассажиров.

- `Stop.get_new_passengers()` — генерация прибывших пассажиров по интенсивности и Δt
- `Stop.record_waiting()` — запись состояния очереди для построения графиков
- `Stop.add_waiting_time()` — учёт времени ожидания для посаженных пассажиров
- `Stop.log_event()` — запись события прибытия трамвая

### 4.3 models/route.py

Ядро симуляции одного прогона маршрута (только fwd **или** только bwd).

- `RouteConfig` — датакласс с полной конфигурацией маршрута, загружается из JSON
- `TripSchedule` — эталонный рейс: `trip_id`, `departure_from_depot`, `stop_times`
- `RouteStats` — агрегированная статистика маршрута: пассажиры, км, отклонения загруженности
- `Route._schedule_dispatcher()` — выпускает трамваи по расписанию, пропускает рейс при отсутствии свободного трамвая
- `Route._tram_run()` — последовательный проезд всех остановок, после финальной — кладёт трамвай в `done_store`
- `Route._arrive_at_stop()` — обработка прибытия: высадка, генерация, посадка, логирование с `trip_id` и `delay_min`
- `Route._calculate_travel_time()` — расчёт времени в пути с учётом `road_loads` и случайности

### 4.4 simulation/multi_route.py

Оркестратор мультимаршрутной симуляции.

- `TramPair` — связывает fwd/bwd маршруты общим пулом трамваев через три Store
- `MultiRouteSimulation.__init__()` — принимает `route_pairs` (dict) и `tram_counts` (list)
- `MultiRouteSimulation.from_params()` — фабричный метод для NSGA-II оптимизатора
- `MultiRouteSimulation.run()` — запуск симуляции, сохранение логов и графиков
- `MultiRouteSimulation.get_objectives()` — возвращает `(avg_waiting_time, total_tram_km, schedule_mae)` для оптимизатора NSGA-II.    Минимизируются все три:
    - `avg_waiting_time` — среднее время ожидания пассажиров (мин), взвешенное по числу обслуженных
    - `total_tram_km` — суммарный пробег всего парка (прокси эксплуатационных затрат)
    - `schedule_mae` — среднее абсолютное отклонение от расписания по всем маршрутам (мин)
- `MultiRouteSimulation.get_full_stats()` — полная статистика по всем маршрутам, включая `schedule_mae_min` по каждому направлению отдельно

### 4.5 simulation/runner.py

CLI-точка входа.

- Принимает номера маршрутов через `--routes`
- Принимает количество трамваев через `--trams` (одно значение на все, или по одному на каждый маршрут)
- Автоматически находит `fwd`/`bwd` конфиги по паттерну `route_{id}_{dir}_config.json`
- При отсутствии любого конфига — выводит список и прерывает запуск
- Поддерживает флаги `--no-plots`, `--no-logs`, `--configs-dir`

### 4.6 visualization.py

Модуль построения аналитических графиков (8 типов). Автоматически определяет `global_stop_id` и отображает локальные номера 1..N.

**Графики очередей:**
- `plot_waiting_passengers()` — динамика очередей на каждой остановке (субплоты)
- `plot_waiting_by_hour()` — среднее число ожидающих по часам (по остановкам)
- `plot_heatmap()` — тепловая карта `YlOrRd` (остановки × часы)

**Графики загруженности трамваев:**
- `plot_utilization()` — загруженность и количество рейсов по трамваям
- `plot_tram_utilization_by_hour()` — почасовая загруженность парка с часами пик

**Графики отклонений от расписания:**
- `plot_delay_by_stop()` — барчарт с ±σ: где маршрут систематически опаздывает
- `plot_delay_by_hour()` — линейный график с тенью: влияние пробок по часам суток
- `plot_delay_heatmap()` — тепловая карта `RdBu_r` (остановки × часы): красный = опоздание, синий = опережение

Все PNG сохраняются с разрешением 150 DPI.

### 4.7 logger.py

Система детального логирования.

- `save_tram_log()` — CSV для каждого трамвая: `trip_id`, `time_min`, `planned_time`, `delay_min`, `stop_id`, `direction`, пассажиропоток, загруженность
- `save_all_trams()` — сохраняет логи всех трамваев маршрута
- `create_summary()` — сводная `trams_summary.csv`: добавлены `avg_delay_min`, `max_delay_min`
- `save_schedule_deviations()` — отдельный `schedule_deviations.csv` со всеми отклонениями (удобно для pandas-анализа)

## 4.8 clear_script.ipynb

Ноутбук для конвертации реальных Excel-выгрузок датчиков пассажиропотока в конфиги модели. Покрывает маршруты 20, 48, 55 в обоих направлениях.

- Фильтрация «сломанных» датчиков (отсечение выбросов >150 пасс., удаление нулевых рейсов)
- Восстановление геометрии маршрута: эталонная последовательность остановок, расстояния из медианного времени хода
- **Фильтрация аномальных замеров времени хода:** принимаются только значения от 3.0 до 15.0 мин — отсечка физически невозможных перегонов из-за глюков датчиков. При отсутствии валидных замеров используется фолбэк 3.0 мин
- Назначение `global_stop_id` с префиксами по маршруту (20→20001..., 48→48001..., 55→55001...)
- Поддержка `STOP_ID_OVERRIDES` для физически совпадающих остановок разных маршрутов
- Генерация реального расписания (`schedule`) с плановыми временами, откалиброванными под физику симуляции: эффективная скорость с учётом средней дорожной нагрузки, время стоянки, посадки/высадки и буфер неравномерности
- Расчёт интенсивности с усреднением по числу валидных дней (мин. `MIN_DAYS = 3`)
- Интерполяция пропущенных часов (линейная внутри дня, затухание на краях)
- Экспорт готовых `.json` в `configs/`

### 4.9 generate_config.py

Генератор полностью синтетических конфигураций:

- Быстрое создание тестовой среды без реальных данных
- Поддерживает `route_id`, `stop_ids`, `operation_start_hour = 5`

## 5. ФОРМАТ ВХОДНЫХ ДАННЫХ

### JSON-конфигурация маршрута

```json
{
  "route_id": "20_fwd",
  "stop_ids": [20001, 20002, "..."],
  "stop_number": 33,
  "distance": [, , "..."],
  "intensity": [
    [20001, 7, 45.3], [20001, 8, 62.1], "..."
  ],
  "schedule": [
    {
      "trip_id": 1,
      "departure_from_depot": 321,
      "stops": [, [20002, 330.1], "..."]
    },
    {
      "trip_id": 2,
      "departure_from_depot": 329,
      "stops": [, [20002, 338.1], "..."]
    }
  ],
  "road_loads": [[0, 0.1], [5, 0.2], [7, 0.7], [8, 0.9], [17, 0.9], [20, 0.4], [23, 0.1]],
  "flow_speed": 15,
  "peak_stop": 20017,
  "tram_capacity": 150,
  "depot_to_first_stop": 8.0,
  "min_rest_time": 15.0,
  "simulation_hours": 24,
  "acceleration_time": 0.5,
  "stop_time": 1.0,
  "turnaround_time": 2.0,
  "target_utilization": 0.75,
  "random_seed": 42
}
```

> **Важно:** поле `tram_count` из конфига **удалено**. Количество трамваев задаётся через CLI (`--trams`) или напрямую в `MultiRouteSimulation`.

### Описание ключевых полей:

| Параметр | Тип | Описание |
| :-- | :-- | :-- |
| `route_id` | String | Идентификатор направления, например `"20_fwd"` |
| `stop_ids` | Array | Упорядоченный список `global_stop_id` остановок маршрута |
| `distance` | Array | `[[global_stop_id, метров_от_предыдущей], ...]`, первая = 0 |
| `intensity` | Array | `[[global_stop_id, час, пасс/час], ...]` — все 24 часа |
| `schedule` | Array | Список рейсов с `trip_id`, `departure_from_depot` и плановыми временами по остановкам (минуты от полуночи) |
| `road_loads` | Array | `[[час, коэффициент 0–1], ...]` — загруженность дорог |
| `flow_speed` | Integer | Базовая скорость (км/ч) при нулевой загруженности |
| `peak_stop` | Integer | `global_stop_id` самой загруженной остановки |
| `tram_capacity` | Integer | Вместимость трамвая (пасс.) |
| `depot_to_first_stop` | Float | Время выезда из депо до первой остановки (мин) |
| `min_rest_time` | Float | Минимальный отдых водителя между рейсами (мин) |
| `simulation_hours` | Integer | Длительность симуляции (часов) |
| `turnaround_time` | Float | Время разворота на конечной (мин) |
| `target_utilization` | Float | Целевая загруженность вагона (доля, обычно 0.75) |
| `random_seed` | Integer | Зерно для воспроизводимости |

### Глобальные ID остановок:

Каждая остановка имеет уникальный `global_stop_id` в формате `{маршрут}{локальный_номер}`:

- маршрут 20 → 20001..20033
- маршрут 48 → 48001...
- маршрут 55 → 55001...

В графиках и логах отображаются **локальные номера 1..N** для читаемости. Физически совпадающие остановки разных маршрутов задаются вручную через `STOP_ID_OVERRIDES` в `clear_script.ipynb`.

## 6. РЕЗУЛЬТАТЫ РАБОТЫ

### 6.1 Консольный вывод


## 7. ЗАПУСК И ИСПОЛЬЗОВАНИЕ

### 7.1 Установка зависимостей

```bash
pip install -r requirements.txt
```


### 7.2 Генерация конфигураций из реальных данных

Поместите Excel-файлы с датчиков в `data_sensors/` и запустите все ячейки `clear_script.ipynb`. Готовые конфиги появятся в `configs/`.

### 7.3 Генерация синтетических конфигураций

```bash
python generate_config.py
```


### 7.4 Запуск симуляции

```bash
# Все три маршрута, по 30 трамваев на каждый (дефолт)
python -m simulation.runner --routes 20 48 55

# Явное задание количества трамваев на каждый маршрут
python -m simulation.runner --routes 20 48 55 --trams 25 30 35

# Одно значение — применяется ко всем маршрутам
python -m simulation.runner --routes 20 48 55 --trams 30

# Без графиков (быстрее)
python -m simulation.runner --routes 20 48 55 --no-plots

# Без CSV-логов
python -m simulation.runner --routes 20 48 55 --no-logs

# Нестандартная папка с конфигами
python -m simulation.runner --routes 20 48 --configs-dir data/configs
```

### 7.5 Анализ результатов

Результаты сохраняются в `outputs/run_ГГГГ-ММ-ДД_ЧЧ-ММ-СС/`. Каждый запуск создаёт отдельную папку — это позволяет сравнивать конфигурации между собой.

Для быстрого анализа отклонений в pandas:

```python
import pandas as pd

df = pd.read_csv("outputs/run_.../logs/20/schedule_deviations.csv")
print(df.groupby("stop_id")["delay_min"].agg(["mean", "std", "max"]))
```
